CREATE TABLE
    users (
        user_id int GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
        username varchar(75) NOT NULL UNIQUE,
        email varchar(260) NOT NULL UNIQUE,
        firstname varchar(255) NOT NULL UNIQUE,
        lastname varchar(255) NOT NULL UNIQUE
    );


FOCUS: user_id int GENERATED ALWAYS AS IDENTITY PRIMARY KEY,

This single line is packed with "rules" that tell PostgreSQL exactly how to handle the most important column in your table. It’s essentially the **Social Security Number** for your data rows.

Here is the breakdown of each part:
---

1. `user_id`
This is simply the **name** of the column. You can call it whatever you want, but `user_id` is the industry standard for identifying a specific user.


2. `int`
This is the **data type**. It tells Postgres that this column can only hold whole numbers (integers). An `int` in Postgres is 4 bytes, which covers a range from roughly **-2.1 billion to +2.1 billion**. (If you expect more than 2 billion users, you'd use `bigint`!).


3. `GENERATED ALWAYS AS IDENTITY`
This is the "magic" part. It tells Postgres to automatically manage the numbers for you (1, 2, 3, 4...).

* **The "Identity" part:** It creates an internal "sequence" (a counter) that increments every time you add a new row.
* **The "ALWAYS" part:** This is a strict safety feature. It tells Postgres: *"Never let a human manually type a number into this column."* If you try to insert a specific ID manually, Postgres will throw an error.
* *Note:* There is also `GENERATED BY DEFAULT`, which allows humans to override the number if they want to, but `ALWAYS` is generally safer for preventing data mess-ups.



4. `PRIMARY KEY`
This is the most important constraint in any database table. It does two things:

1. **Unique:** No two users can ever have the same `user_id`.
2. **Not Null:** This column can never be empty. Every row *must* have an ID.
3. **Indexing:** It automatically creates an "Index," which makes searching for a specific user by their ID lightning-fast, even if you have millions of rows.



---

Why use this instead of `SERIAL`?

If you've looked at older tutorials, you might have seen people use `user_id SERIAL PRIMARY KEY`.

While `SERIAL` still works, `GENERATED ALWAYS AS IDENTITY` is the **modern, SQL-standard way** of doing things. It’s more robust, handles permissions better, and is easier to manage if you ever need to move your database to a different system (like SQL Server or Oracle).

> **Quick Summary:** You’re telling Postgres: *"Create a column called user_id for numbers. Automatically assign the next number in line every time I add a user, don't let me manually mess with those numbers, and make sure every ID is unique so I can use it to find people quickly."*
